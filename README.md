# EC2 CloudWatch Monitoring and Alerts

Terraform configuration for a small AWS monitoring environment. It creates a public Ubuntu EC2 instance, installs the Amazon CloudWatch Agent, collects authentication logs, provides a CloudWatch dashboard, and sends email notifications for defined CPU and security alarms.

![Architecture](docs/architechure.png)

## What is deployed

- A VPC (`10.0.0.0/16`) with one public subnet (`10.0.1.0/24`), an internet gateway, and a default route to the internet.
- An Ubuntu 24.04 EC2 instance with a public IP address and an IAM instance profile that grants `CloudWatchAgentServerPolicy`.
- An SSH key pair generated by Terraform; its private key is written to `terraform/ssh-key.pem` with `0600` permissions.
- A CloudWatch Logs group, `/aws/ec2/auth`, retaining logs for 14 days.
- An SNS topic, `critical-alerts`, with an email subscription.
- CloudWatch alarms for CPU, memory, disk, unauthorized SSH login attempts, and `sudo` usage.
- A CloudWatch dashboard with CPU, memory, disk, network, security, and EC2 status-check widgets.

## Prerequisites

- An AWS account and credentials configured locally (for example, with `aws configure` or environment variables).
- Terraform 1.5 or newer.
- Permission to create EC2, VPC, IAM, CloudWatch, and SNS resources in the selected AWS Region.

The default region is `eu-north-1` and the default instance type is `t3.micro`. Change these in `terraform/terraform.tfvars` or pass values with `-var`.

## Deploy

1. Review `terraform/terraform.tfvars` and set the alert email value if required.

2. Review the email endpoint in `terraform/sns.tf`. The SNS subscription currently uses a literal email address, so changing `email_address` in `terraform.tfvars` alone does **not** change the subscription. Update `sns.tf`, or refactor its `endpoint` to use `var.email_address`, before applying.

3. Initialize, validate, and apply the Terraform configuration:

   ```bash
   cd terraform
   terraform init
   terraform validate
   terraform plan
   terraform apply
   ```

4. Confirm the SNS subscription from the verification email AWS sends. Alerts cannot be delivered until this is confirmed.

5. Get the connection command:

   ```bash
   terraform output -raw ssh_command
   ```

## Configure log and host metrics collection

The EC2 user data installs the CloudWatch Agent but does not configure or start it. Copy the repository configuration to the instance and start the agent after connecting over SSH:

```bash
scp -i terraform/ssh-key.pem cloudwatch/amazon-cloudwatch-agent.json \
  ubuntu@<INSTANCE_PUBLIC_IP>:/home/ubuntu/

ssh -i terraform/ssh-key.pem ubuntu@<INSTANCE_PUBLIC_IP>
sudo cp /home/ubuntu/amazon-cloudwatch-agent.json \
  /opt/aws/amazon-cloudwatch-agent/etc/amazon-cloudwatch-agent.json
sudo /opt/aws/amazon-cloudwatch-agent/bin/amazon-cloudwatch-agent-ctl \
  -a fetch-config -m ec2 \
  -c file:/opt/aws/amazon-cloudwatch-agent/etc/amazon-cloudwatch-agent.json -s
```

The included JSON collects `/var/log/auth.log` into `/aws/ec2/auth`. To make the memory and disk alarms operational, extend `cloudwatch/amazon-cloudwatch-agent.json` with a `metrics.metrics_collected` section for `mem_used_percent` and `disk_used_percent`; the current configuration only configures logs.

## Alerts and dashboard

| Alarm | Metric | Trigger |
| --- | --- | --- |
| `HighCPUUsage` | `AWS/EC2` `CPUUtilization` | Average above 85% for 5 minutes |
| `HighMemoryUsage` | `CWAgent` `mem_used_percent` | Average above 85% for 5 minutes |
| `HighDiskUsage` | `CWAgent` `disk_used_percent` | Average above 90% for 5 minutes |
| `UnauthorizedLoginAttempts` | `SecurityMetrics` `UnauthorizedLoginAttempts` | Sum of 5 or more for 5 minutes |
| `SudoCommandsDetected` | `SecurityMetrics` `SudoCommands` | Sum above 10 for 5 minutes |

All alarms publish to the `critical-alerts` SNS topic. The dashboard is named `dashboard-1` and can be opened in the CloudWatch console after the apply completes.

The dashboard includes built-in EC2 CPU/network/status metrics and custom agent/security metrics. The disk widgets and alarm are tied to the current Ubuntu root-disk dimensions (`/`, `nvme0n1p16`, `ext4`); verify these if the AMI or block-device layout changes.

## Testing alerts

After the agent and SNS subscription are working:

```bash
# Copy and run the CPU test on the EC2 instance
scp -i terraform/ssh-key.pem scripts/generate-cpu-load.sh \
  ubuntu@<INSTANCE_PUBLIC_IP>:/home/ubuntu/
ssh -i terraform/ssh-key.pem ubuntu@<INSTANCE_PUBLIC_IP> \
  'bash /home/ubuntu/generate-cpu-load.sh'

# From a machine that can reach the EC2 instance: generate invalid SSH logins
# Replace SERVER_IP with the instance public IP.
bash scripts/simulate-login-attempts.sh
```

`generate-cpu-load.sh` installs `stress` and uses four CPU workers for 300 seconds. `simulate-login-attempts.sh` makes ten SSH attempts with a fake username, which should create `Invalid user` records in `auth.log`.

## Security notes

- The security group currently permits SSH from `0.0.0.0/0`. Restrict port 22 to a trusted CIDR before using this outside a short-lived demo.
- `terraform/ssh-key.pem` is a private key generated during apply. Keep it private; do not commit or share it.
- Avoid committing real email addresses or Terraform state files. State can contain infrastructure metadata and generated key material.

## Tear down

Destroy all resources when finished to avoid AWS charges:

```bash
cd terraform
terraform destroy
```
